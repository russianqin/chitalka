## 一、最新变化（v3.0）

### Bug 修复（2026-05-10）

**问题 1：书签功能完全失效**
- **现象**：点击「🔖 书签」按钮无反应，控制台无报错
- **根因**：`init()` 函数调用了 `bindBookEvents()`，但忘记调用 `bindEvents()`
- **后果**：所有事件绑定代码（包括书签、字体控制等）都不执行
- **修复**：在 `init()` 中添加 `bindEvents()` 调用

**问题 2：`handleFontBold is not defined` 报错**
- **现象**：控制台报错 `Uncaught ReferenceError: handleFontBold is not defined`
- **根因**：同上，`bindEvents()` 未执行，导致 `window.handleFontBold` 未定义
- **影响**：加粗按钮（B）无法使用，字号、行距滑块也可能异常
- **修复**：同上，`init()` 中调用 `bindEvents()`

**问题 3：重复的 `loadFromStorage()` 函数定义**
- **现象**：多书籍功能失效
- **根因**：文件中两次定义 `loadFromStorage()`，第二个（旧版，无多书籍支持）覆盖第一个
- **修复**：删除第 1162-1197 行的重复定义

**验证结果**：修复后，书签功能、字体控制功能均恢复正常。

### 多书籍支持

新增多书籍管理能力，告别「一本书读完再换」的低效流程：

**核心能力**

- 同时导入并管理多本书籍，生词本全局共用
- 点击顶部 Header 中间的「书名」按钮，弹出书籍列表下拉菜单
- 下拉菜单显示所有已导入书籍（书名、字数、最后阅读时间）
- 点击书籍名称瞬间切换，自动恢复阅读进度
- 底部提供「📂 导入新书」按钮，随时添加新书籍
- 支持移除书籍（至少保留一本，防止误操作）

**数据迁移**

- 首次启动自动检测旧版 `chitalka_reading_v1` 数据
- 自动迁移为单书籍存入新结构，无需手动操作
- 迁移前不改变旧数据，确认成功后才写入，零丢失风险
- 旧书签自动关联迁移后的书籍 ID

**数据存储变更**

| 数据 | 旧版 Key | 新版 Key | 说明 |
|------|-----------|----------|------|
| 书籍列表 | — | `chitalka_books` | 数组，存储所有书籍的 id / 文件名 / 全文 / 阅读进度 |
| 当前书籍 | `chitalka_reading_v1` | `chitalka_active_book_id` | 当前正在读的书籍 ID |
| 生词本 | `chitalka_vocab_v4` | 不变 | 全局共用，所有书籍共享 |
| 书签 | `chitalka_bookmarks` | 不变 | 每条书签新增 `bookId` 字段，关联书籍 |

**书签增强**

- 书签列表显示所属书籍名称
- 点击书签时，若所属书籍非当前书籍，自动切换书籍后再跳转
- 书签按添加时间倒序排列

> 💡 **设计决策**：生词本全局共用，因为同一个读者在不同书籍中遇到的生词应该累积，而不是按书隔离。这符合语言学习的实际情况。

## 二、软件功能

### 2.1 阅读器

**文本加载**

- 支持拖放 `.txt` / `.text` / `.md` 等纯文本文件到页面
- 也可点击上传按钮选择文件
- 支持超大文件（内容全部存入内存，无额外依赖）
- **新增**：支持同时导入多本书籍，通过下拉菜单切换

**段落结构**

- 原始段落间距完整保留
- 俄语词以浅绿色背景高亮显示
- 单词点击可查词并收录

**阅读控制栏**（顶部工具栏）

| 控件 | 功能 |
|------|------|
| 书籍切换 | 点击书名按钮弹出下拉列表，快速切换书籍 |
| 字体切换 | 衬线体 / 无衬线体 |
| 字号滑块 | 14px ~ 28px，拖动实时预览 |
| 行距滑块 | 1.2 ~ 2.5，步进 0.1，默认 2.0 |
| 加粗按钮 | 一键切换正文加粗，再点取消 |
| 书签按钮 | 打开书签管理面板，支持添加、跳转、删除书签 |
| 模式切换 | 📖 普通 / 🔬 透析，透析模式限制生词查阅频率 |
| 字数统计 | 显示文档总词数（非生词数） |

> 字体、字号、加粗状态自动存入 localStorage，刷新页面保持。

---

### 2.2 AI 智能收录 + 翻译

**单词收录**

1. 阅读时点击任意俄语单词
2. 弹出浮动工具栏，显示当前单词
3. 点击 **🤖 AI 收录** 按钮
4. 弹出收录面板，DeepSeek AI 并行生成以下内容：

| AI 生成字段 | 说明 |
|-------------|------|
| 原型（词典形式） | 动词变位 / 名词变格后的原形 |
| 英文释义 | 单词原型对应的英文翻译（简洁，1~3 个词） |
| 中文释义 | 精确释义，含语境义 |
| 词根词缀分析 | 前缀 / 词根 / 后缀逐层拆解 |
| 记忆技巧 | 联想记忆、同源词提示 |
| 原文例句 | 从文档中提取的完整句子 |
| 语法标注 | 标注词性、体、时、数、格等 |

5. 用户可编辑所有字段，确认后保存

**设置项**

- DeepSeek API Key（存储在浏览器本地，不上传）
- 模型选择：`deepseek-chat`（推荐）或 `deepseek-coder`
- 获取地址：[platform.deepseek.com](https://platform.deepseek.com)

**多词翻译**

1. 选中一段俄语文本（≥2 个词）
2. 弹出浮动工具栏，显示 **🌐 翻译** 按钮
3. 点击后调用 AI 翻译，直接展示中文译文
4. 无需保存，专注当前阅读理解

**列表模式**

- 显示所有已收录单词（**全局共用，跨书籍累积**）
- 顶部搜索框：支持按单词、释义全文搜索
- 单词按收录时间倒序排列
- 点击任意卡片展开详情

**详情面板**

- 显示完整学习资料（释义、词根、记忆技巧、例句、语法）
- 可手动编辑或删除该词条

### 2.4 导入 / 导出

**导入**（设置页面 → 数据管理 → 导入词汇）

- 支持 `.csv` 和 `.tsv` 文件
- 自动识别分隔符（逗号 / 制表符）
- 智能匹配列名（识别"单词/слово/word"、"释义/trans"等常见列名）
- 导入前显示预览，确认后执行
- 自动跳过重复单词和空行

**导出**（点击「💾 备份」按钮弹出导出面板）

| 格式 | 说明 |
|------|------|
| TXT（文本） | 可选择日期范围，包含完整字段（收录日期、单词、原型、英文释义、释义、词根分析、记忆技巧、原文例句），同时复制到剪贴板 |
| JSON 备份 | 支持按日期筛选，包含所有词汇和设置，可完整恢复 |

### 2.5 书签功能

**添加书签**

- 加载文档后，点击阅读工具栏的「🔖 书签」按钮
- 点击「➕ 添加书签」记录当前阅读位置
- 自动记录：添加日期（YYYY-MM-DD）、精确到秒的时间戳（HH:mm:ss）、当前阅读进度百分比
- **新增**：书签关联所属书籍 ID，跨书籍跳转自动切换

**书签列表**

- 显示所有书签：书名、添加日期、添加时间、阅读进度百分比
- 点击任意书签条目跳转至对应位置（自动切换书籍）
- 点击删除按钮移除书签

**导出日期筛选**

- 点击「💾 备份」按钮弹出导出面板
- 可选择开始/结束日期，留空则导出全部
- 提供「今天」「本周」快捷按钮
- 默认按收录日期升序导出

### 2.7 透析模式

透析模式是一种**控制生词查阅频率**的学习模式，目的是促使读者尽量通过上下文猜测词义，而不是遇词就查。

**两种模式对比**

| 模式 | 按钮 | 行为 |
|------|-------|------|
| 普通模式 | 📖 普通 | 点击任意单词均可正常查阅/收录 |
| 透析模式 | 🔬 透析 | 点击未高亮词受 1 分钟间隔限制 |

**透析模式规则**

1. 点击**已高亮词**（已收录）：直接弹出详情，不受限制
2. 点击**未高亮词**（未收录）：检查距上次成功收录生词是否满 1 分钟
   - 间隔 ≥ 1 分钟：正常弹出 AI 收录面板
   - 间隔 < 1 分钟：拒绝查阅，Toast 提示「透析模式下查阅生词间隔不少于1分钟」
3. **计时基准**：生词成功保存到生词本时重置计时器（不是点击查阅时）

**计时器持久化**

- 页面刷新后，自动读取生词本中最新词条的 `savedAt` 字段恢复计时基准
- 若生词本为空，则透析模式下首次点击未高亮词不受限制

> 适合有一定俄语基础、希望锻炼上下文猜词能力的用户。

## 六、开发手记：为什么自己用很重要

> 自己是最了解自己产品的人。自己在使用中，能更高效地发现不足，并及时完善。

Читалка Pro 的每一个版本，几乎都来自一个很简单的循环：

1. **做一个功能** → 自己用起来
2. **遇到别扭的地方** → 记下来
3. **立刻改** → 再继续用

这次 v3.0 的「多书籍支持」也是一个典型例子：

- 之前每次换书都要重新导入、丢失阅读进度，非常低效
- 自己用的时候一直觉得别扭，终于决定一次性解决
- 实现后发现数据迁移是个难点，反复测试确保不丢数据

如果等产品「写完再测」，这类实际需求很容易漏掉。而自己用自己做的工具，每一次使用都是一次需求收集，每一次别扭都是一次迭代信号。

**这就是为什么 Читалка Pro 的迭代速度这么快**——不是因为代码写得有多好，而是因为用户和开发者是同一个人。
